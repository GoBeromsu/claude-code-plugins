# 문서 구조 원칙 (Document Structure Principles)

좋은 문서 구조는 독자가 목적을 달성하는 데 드는 탐색 비용을 최소화한다. 아래 원칙은 Toss 기술 문서 가이드에서 추출한 페이지·섹션 단위 구조 기준이다.

---

## 1. 한 페이지, 한 주제 (One Page, One Topic)

페이지 하나는 독자의 질문 하나를 답해야 한다. 헤딩 계층이 H4를 초과하거나 페이지가 복수의 목표를 다루면 분리 신호다.

**분리 기준**:
- 헤딩 깊이가 H4(`####`)를 초과할 때
- 페이지가 두 가지 이상의 독자 목적을 동시에 다룰 때 (예: 튜토리얼 + API 레퍼런스 혼합)
- 독자가 서로 다른 배경 지식으로 이 페이지에 도착할 때

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "인증 가이드"에 개념 설명 + 설치 튜토리얼 + API 레퍼런스가 모두 포함 | 개념 설명 → explanation 페이지 / 설치 튜토리얼 → learning 페이지 / API → reference 페이지로 분리 |

---

## 2. 가치 먼저 (Value First)

독자는 "왜 읽어야 하는가"를 먼저 알아야 계속 읽는다. 기능(feature)보다 가치(value)를 앞세운다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "이 가이드는 Docker Compose 설정 방법을 설명합니다." | "Docker Compose를 사용하면 여러 서비스를 하나의 명령으로 실행할 수 있습니다." |
| "이 API는 `createToken` 함수를 제공합니다." | "이 API로 사용자 세션을 안전하게 관리할 수 있습니다." |

**패턴**: 첫 단락 = "~를 사용하면 [가치]를 얻을 수 있다."

---

## 3. 제목 가이드라인 (Title Guidelines)

좋은 제목은 독자가 검색했을 법한 핵심 키워드를 앞에 배치하고, 짧고 일관된 형식을 유지한다.

**규칙**:
- 핵심 키워드를 제목 앞에 배치 (예: "Docker로 FastAPI 컨테이너화")
- 30자 이내
- 질문형 금지 (예: "JWT가 무엇인가요?" → "JWT 인증 원리")
- 문서 유형에 따라 일관된 형식 선택:
  - 학습/튜토리얼: 동사형 — "Docker로 FastAPI 컨테이너화하기"
  - How-to: 동사형 — "GitHub Actions로 Node.js 배포 자동화"
  - 참조: 명사형 — "`createToken` API"
  - 설명: 명사형 — "JWT 인증 원리"

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "인증에 대해서 알아볼게요 — JWT 토큰이 무엇인지 살펴보겠습니다" | "JWT 인증 원리" |
| "새로운 기능을 사용하는 방법에 대한 가이드" | "다크 모드 활성화 가이드" |

---

## 4. 개요 단락 필수 (Mandatory Overview)

모든 페이지는 첫 번째 헤딩 앞에 개요 단락을 포함해야 한다. 개요 단락은 이 페이지에서 무엇을 다루는지, 독자가 무엇을 얻는지를 1–3 문장으로 요약한다.

**나쁜 예** (개요 없음):
```markdown
# Docker로 FastAPI 컨테이너화하기

## 사전 요구사항
...
```

**좋은 예** (개요 있음):
```markdown
# Docker로 FastAPI 컨테이너화하기

Docker를 사용해 FastAPI 앱을 컨테이너화하면 어떤 환경에서도 동일하게 실행할 수 있습니다.
이 튜토리얼을 따라하면 로컬에서 동작하는 FastAPI 앱을 Docker 이미지로 빌드하고 실행할 수 있습니다.

## 사전 요구사항
...
```

---

## 5. 예측 가능한 구조 (Predictable Structure)

동일 유형의 문서는 같은 헤딩 패턴을 반복해야 한다. 독자가 두 번째 문서를 읽을 때 구조를 예측할 수 있어야 한다.

- 같은 문서 유형 → 같은 헤딩 순서 (reference 파일의 템플릿을 따른다)
- 섹션 순서를 임의로 바꾸지 않는다
- 이 문서에만 있는 특수 섹션은 공통 섹션 뒤에 배치

---

## 6. 종합 주제 개요 페이지 (Overview Page Pattern)

문서 집합의 진입점에는 Overview 페이지를 둔다. Overview 페이지는 하위 페이지 목록과 각 페이지의 한 줄 설명을 제공한다.

```markdown
# [주제] 개요

[이 문서 모음이 다루는 범위와 독자 목적 — 한 단락]

## 포함 문서

| 문서 | 설명 |
|------|------|
| [시작하기](get-started.md) | 설치부터 첫 실행까지 |
| [API 레퍼런스](reference/) | 모든 함수·설정 스펙 |
| [트러블슈팅](troubleshooting.md) | 자주 발생하는 오류 해결 |
```
