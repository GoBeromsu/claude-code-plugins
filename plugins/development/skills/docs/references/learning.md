# 학습 중심 문서 Guide (Learning / Tutorial)

독자가 처음부터 끝까지 막힘없이 따라하며 새로운 기술·도구를 배울 수 있도록 돕는 문서.

## 두 가지 하위 유형

| 유형 | 특징 |
|------|------|
| **시작하기 (Getting Started)** | 전체 흐름과 핵심 개념 이해. 설치·설정 중심 |
| **튜토리얼 (Tutorial)** | 명확한 목표·결과물이 있는 단계별 실습 |

## 작성 원칙

1. **학습 목표를 서두에 명시** — 독자가 이 문서를 읽고 무엇을 할 수 있는지 한 문장으로
2. **단계별 구성** — 목표 → 사전요구사항 → 단계별 실습 → 최종 결과 확인
3. **실행 가능한 예제** — 복사해서 바로 실행되는 코드만 포함. 검증 필수
4. **성공 경로에 집중** — 예외·부가 설명은 FAQ 또는 접어두기로 분리

## 템플릿

```markdown
# [튜토리얼 제목]

## 목표

[이 문서를 따라하면 독자가 달성할 결과 — 한 문장]

## 사전 요구사항

- [필요한 사전 지식 또는 설치할 도구]

## [첫 번째 단계 제목]

[작업 설명]

\`\`\`언어
// 실행 가능한 예제 코드
\`\`\`

[실행 후 기대 결과 — 독자가 성공 여부를 스스로 확인할 수 있어야 함]

## [두 번째 단계 제목]

...

## 최종 결과 확인

[모든 단계를 완료했을 때 보여야 할 결과]
```

## 안티패턴 (Anti-Patterns)

**안티패턴 1: 메타 진술로 시작하기**

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "이 문서는 Docker로 FastAPI를 컨테이너화하는 방법을 설명합니다." | "Docker를 사용해 FastAPI 앱을 어디서나 동일하게 실행할 수 있습니다. 이 튜토리얼을 따라하면 로컬 앱을 이미지로 빌드하고 실행할 수 있습니다." |

문서 자신을 소개하는 문장("이 문서는 ~을 설명합니다")을 개요 단락으로 교체한다. 개요는 독자가 얻는 가치를 직접 서술한다.

**안티패턴 2: 결과 검증 없는 단계**

각 단계 완료 후 독자가 성공 여부를 스스로 확인할 수 있어야 한다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "다음 명령으로 Docker 이미지를 빌드합니다." | "다음 명령으로 Docker 이미지를 빌드합니다. 완료되면 `docker images` 명령을 실행해 `fastapi-app` 이미지가 목록에 있는지 확인합니다." |

## 체크리스트

- [ ] 학습 목표("목표" 섹션)를 첫 번째 헤딩 앞에 명시했는가?
- [ ] 개요 단락이 문서 자신을 소개하는 대신 독자가 얻는 가치를 설명하는가?
- [ ] 예제 코드를 복사해 실행하면 오류 없이 동작하는가? (실제 검증 필수)
- [ ] 각 코드 블록 아래에 기대 출력 또는 확인 방법이 있는가?
- [ ] 단계별 구성 (목표 → 사전요구사항 → 단계 → 최종 결과 확인)
- [ ] 각 단계 완료 시 독자가 성공 여부를 스스로 확인할 수 있는가?
