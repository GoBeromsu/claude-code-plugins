# 문제 해결 문서 Guide (How-to & Troubleshooting)

독자가 직면한 구체적인 문제를 빠르게 해결하도록 돕는 문서.

## 두 가지 하위 유형

| 유형 | 특징 |
|------|------|
| **How-to 가이드** | 특정 기능 구현을 위한 단계별 절차. 목표 달성 중심 |
| **트러블슈팅 문서** | 이미 발생한 오류 진단·해결. 디버깅 과정 중심 |

## 작성 원칙

1. **문제 상황 먼저 정의** — 원인과 현상을 구분해서 설명. 에러 메시지·로그 예시 포함
2. **즉시 적용 가능한 해결 방법** — 코드, 명령어, 설정값 포함. 해결 원리도 한 줄 언급
3. **환경별 차이 고려** — OS, 라이브러리 버전 차이를 명시
4. **해결 후 확인 방법 포함** — 정상 동작을 독자가 스스로 검증할 수 있어야 함

## 템플릿 — How-to 가이드

```markdown
# [작업 제목]

[이 가이드의 목표와 달성했을 때 얻는 가치 — 한 단락]

## 사전 요구사항

[환경 설정이나 필수 조건. 없으면 생략]

## [첫 번째 단계]

[설명]

\`\`\`언어
// 예제 코드
\`\`\`

## [두 번째 단계]

...

## 결과 확인

[작업 완료 후 정상 동작 확인 방법]
```

```markdown

[문제 상황과 발생 원인 — 한 단락]

[진단 방법]

\`\`\`bash
# 진단 명령어
\`\`\`

## 2. [다음 해결 방법]

[해결 방법]

\`\`\`bash
# 해결 명령어
\`\`\`

## [선택] 환경별 추가 확인

[OS, 버전 차이 등]
```

## 안티패턴 (Anti-Patterns)

**안티패턴 1: 원인과 현상 혼동**

원인과 현상을 구분하지 않으면 독자가 잘못된 해결책을 시도한다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "CORS 오류가 발생합니다. 서버가 응답하지 않아서입니다." | **현상**: 브라우저 콘솔에 `Access-Control-Allow-Origin` 오류가 표시됩니다. **원인**: 서버가 요청 출처를 허용 목록에 포함하지 않았기 때문입니다. |

**안티패턴 2: 여러 해결책의 순서 없음**

트러블슈팅에서 해결책이 여럿이면 가장 흔한 원인을 먼저 제시한다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 원인 A, B, C를 중요도 없이 나열 | "## 1. 가장 흔한 원인: 환경 변수 누락 (전체 케이스의 70%)" 순으로 정렬 |

## 체크리스트

- [ ] 현상(증상)과 원인을 명확히 구분해 설명했는가?
- [ ] 여러 해결책이 있다면 가장 흔한 원인부터 순서대로 정렬했는가?
- [ ] 각 해결책에 실행 가능한 명령어 또는 코드가 포함됐는가?
- [ ] 환경별 차이(OS, 라이브러리 버전)를 명시했는가?
- [ ] "결과 확인" 섹션에서 독자가 정상 동작을 스스로 검증할 수 있는가?
