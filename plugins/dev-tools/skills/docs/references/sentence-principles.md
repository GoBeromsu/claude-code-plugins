# 문장 작성 원칙 (Sentence-Level Writing Principles)

좋은 문장은 구조가 명확하고, 핵심만 담으며, 일관된 언어를 사용한다. 아래 다섯 가지 원칙은 Toss 기술 문서 스타일에서 추출한 문장 단위 기준이다.

---

## 1. 능동 주어 (Active Subject)

행위자를 명확히 드러내라. 수동태는 책임 주체를 감춘다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 설정이 완료됩니다. | SDK가 설정을 완료합니다. |
| 오류가 반환됩니다. | 서버가 400 오류를 반환합니다. |
| 토큰은 검증되어야 합니다. | 서버가 요청마다 토큰을 검증합니다. |

**규칙**:
- "~됩니다" 패턴을 보면 행위자를 찾아 주어로 세운다
- "~하면 됩니다" → "~하십시오" (명령형이 더 직접적)
- 주어가 "시스템", "앱" 등 모호한 명사면 구체적인 컴포넌트로 교체

---

## 2. 간결성 (Conciseness)

문장 하나에 핵심 아이디어 하나만 담아라. 메타 담론과 명사화를 제거한다.

### 메타 담론 제거

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 앞서 언급한 것처럼, JWT는 stateless입니다. | JWT는 stateless입니다. |
| 아마 알다시피 React는 Virtual DOM을 사용합니다. | React는 Virtual DOM을 사용합니다. |
| 다음 섹션에서는 설치 방법을 설명합니다. | *(섹션 제목으로 대체)* |

### 명사화 → 동사화

명사형 표현은 동사로 바꾸면 문장이 짧아지고 주어가 생긴다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 데이터의 검증을 수행합니다. | 데이터를 검증합니다. |
| API 키의 발급이 필요합니다. | API 키를 발급받아야 합니다. |
| 설정의 초기화를 진행합니다. | 설정을 초기화합니다. |

### 한 문장 한 아이디어

| 나쁜 예 | 좋은 예 |
|--------|--------|
| JWT는 헤더, 페이로드, 서명의 세 부분으로 구성되며, 서버는 요청마다 서명을 검증하여 토큰의 위변조 여부를 확인합니다. | JWT는 헤더·페이로드·서명 세 부분으로 구성됩니다. 서버는 요청마다 서명을 검증해 위변조를 확인합니다. |

---

## 3. 구체성 (Specificity)

추상 명사 대신 구체적인 명사를 쓰고, 행동 동사로 의미를 명확히 한다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 적절한 설정이 필요합니다. | `timeout`을 30초로 설정합니다. |
| 올바른 방식으로 처리하세요. | `catch` 블록에서 에러를 로깅하고 사용자에게 알림을 표시하세요. |
| 문제가 발생하면 확인하세요. | `npm run test`를 실행해 실패한 테스트를 확인하세요. |

**규칙**:
- "적절한", "올바른", "관련된" 같은 수식어를 보면 구체적인 값으로 교체
- "처리하다", "확인하다"는 더 구체적인 동사로: 로깅하다, 반환하다, 검증하다, 파싱하다

---

## 4. 일관성 (Consistency)

같은 개념은 문서 전체에서 같은 용어로 표현한다.

### 동의어 혼용 금지

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "사용자"와 "유저"와 "클라이언트"가 혼재 | 하나의 용어("사용자")로 통일 |
| "오류", "에러", "에러 메시지"가 섞임 | 문서 전체에서 "오류"로 통일 |

### 약어 첫 등장 풀기

| 나쁜 예 | 좋은 예 |
|--------|--------|
| JWT를 발급합니다. *(첫 등장)* | JSON Web Token(JWT)을 발급합니다. *(이후 "JWT"만 사용)* |
| CI/CD 파이프라인을 구성합니다. | Continuous Integration/Continuous Deployment(CI/CD) 파이프라인을 구성합니다. |

### 외래어 표기 전략 통일

한 문서에서 영문 원어와 한글 음차를 섞지 않는다.

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "token"과 "토큰"이 혼재 | 둘 중 하나로 통일 (기술 문서는 "토큰" 권장) |

## 5. 한국어 자연스러운 표현 (Korean Natural Expression)

번역투와 불필요한 한자어를 제거한다.

### 번역투 제거

| 나쁜 예 (번역투) | 좋은 예 |
|----------------|--------|
| 이것은 … 할 수 있게 해줍니다. | …합니다. / …할 수 있습니다. |
| ~함으로써 성능을 향상시킬 수 있습니다. | ~하면 성능이 향상됩니다. |
| 해당 기능을 통해 사용자는 … | 사용자는 이 기능으로 … |

### 불필요한 한자어 제거

| 나쁜 예 | 좋은 예 |
|--------|--------|
| 상기 방법을 사용하십시오. | 위 방법을 사용하십시오. |
| 하기 사항을 확인하십시오. | 다음 사항을 확인하십시오. |
| 본 문서에서는 … | 이 문서에서는 … |

## 빠른 자가 점검 (Self-Check)

문장을 쓴 뒤 아래를 확인한다:

1. "~됩니다" → 행위자가 주어인가?
2. "앞서", "아마", "다음 섹션에서는" → 삭제할 수 있는가?
3. "의 ~을" 연속 → 동사로 바꿀 수 있는가?
4. 같은 개념을 다른 단어로 불렀는가?
5. 약어를 처음 등장 시 풀어 썼는가?
